P1 — закрыть до анонсаДва пункта. Оба чинятся дёшево, без переписывания.

P1BUG-01api-contract Deprecated-алиасы старых имён тулов не реализованы
Гдеmcp.mcphire.com/sse — register_profile / delete_profile
ЖдалиПо плану Спринта 2 и по обещанию в server.json («30-day deprecation window») старые имена работают как алиасы и возвращают deprecation_warning.
ФактАлиасов нет. tools/call register_profile → жёсткий Unknown tool. Агент со старым кэшем падает без подсказки про новое имя.
{"content":[{"text":"Unknown tool: 'register_profile'"}],"isError":true}
КонтекстДенис сам отмечал «~0 реальных юзеров», поэтому риск для текущей базы низкий — но обещанный в манифесте контракт не выполнен. Чинится регистрацией двух обёрток-алиасов.
▸ нашли независимо Agent A + Agent C
P1BUG-02api-contract У delete_employer_profile нет REST-аналога
Где/api/v1/employer/* — удаление профиля
ЖдалиПо симметрии с кандидатом (DELETE /api/v1/candidate/profile работает) у нового тула должен быть REST-twin.
ФактREST-аналога нет: DELETE /employer/profile, /employer/{id}, POST /employer/delete — все 404. Тул только через MCP. Работодатель на чистом REST-fallback не удалит компанию, хотя кандидат — может.
{"detail":"Not Found"} HTTP:404

P2 — высокий приоритетНе блокируют запуск, но портят опыт первого визита и засоряют данные.

P2BUG-03docs-drift · ux skill.md не предупреждает про draft-gate вакансий
ФактКанон шаг 4 говорит «опубликовано, матчинг сработает сам». Реально вакансия НЕверифицированного работодателя молча уходит в status=draft, невидима на /jobs (страница отдаёт 410). Агент думает, что опубликовал, и ждёт откликов, которых не будет.
post_vacancy: {"status":"draft"} · публичная страница: HTTP:410
▸ нашёл Agent B
P2BUG-04security-hygiene proof_url не валидируется на входе (но реального SSRF нет)
ФактРегистрация принимает любой URL — 169.254.169.254, localhost, file:///etc/passwd — все 201, URL сохраняется как есть.
НоRoot cause проверен в коде: единственный потребитель URL — claim-verifier — фетчит только через safe_fetch, который блокирует приватные/loopback/AWS-метадату/DNS-rebinding (Codex round 1-9). Эксплуатации нет — это гигиена входа и мусор в БД, не дыра. Severity понижен P1 → P2 после проверки.
register: HTTP:201 · safe_fetcher.py PRIVATE_NETS включает 169.254.0.0/16
▸ нашёл Agent C · root cause доуточнён вручную
P2BUG-05registration Нет дедупа — неограниченные дубли профилей
ФактЧетыре регистрации (2 параллельных + 2 последовательных) с идентичным proof_url+email → 4 разных профиля, все 201. Дедупа нет. Rate-limit лишь замедляет. Засоряет матчинг.
P1 / P2 / seq3 / seq4 → 4 distinct profile_id, все HTTP:201
▸ нашёл Agent C
P2BUG-06ux consent-hint врёт при несовпадении фразы
ФактПри несовпадении consent-фразы hint всегда «set flag to true», хотя флаг уже true. Агент будет крутить флаг, а не фразу.
{"error":"CONSENT_PHRASE_MISMATCH...","hint":"set ...to true"} HTTP:400
▸ нашёл Agent B
P2BUG-07registration · security consent_phrase можно опустить (окно открыто бессрочно)
Фактconsent_granted=true + валидный proof_url без consent_phrase → 201. By-design «30-day deprecation window», но окно открыто без срока — позволяет регистрировать без фразы-подтверждения.
▸ нашёл Agent C
P2BUG-08docs-drift proof_url не описан в шагах skill.md
Фактproof_url (обязательное поле) упомянут только в «Error recovery» внизу, не в шагах 1-6. Свежий агент сверху вниз гарантированно ловит 400 на первой регистрации (спасает hint на втором заходе).
HTTP 400 {"error":"proof_url ... is required for claim verification."}

Наблюдение — не баг для этого запуска

NOTENOTE-A1архитектура онбординга Нет веб-онбординга для людей без агента
СутьВесь онбординг — вокруг «отправь агенту строку Read skill.md». Для человека без MCP-клиента веб-формы регистрации нет.
Почему ОККогорта софт-лаунча — только те, кто уже пользуется агентами. Для них MCP-first путь и есть продукт. Веб-форма — продуктовое решение на потом.
ПодстраховкаЕсли в группе 15-20 окажется кто-то без настроенного клиента — дать ему персональную мини-инструкцию «поставь Claude Desktop → вставь конфиг».
▸ Agent D

Что прошло чисто — это работает

  • Rename тулов корректен во всех артефактах (server.json, skill.md, llms.txt) — старых имён в доках нет нигде.
  • delete_employer_profile работает: без confirm → confirm_required, мусорный токен → не 500, happy-path → каскад (deleted_jobs:1), профиль исчез.
  • SSoT skill.md — настоящий единый источник: llms.txt и server.json ссылаются, не дублируют протокол.
  • Rate-limit регистрации = 10/min/IP → 429 + Retry-After (ровно как заявлено).
  • SSRF-guard в safe_fetcher — comprehensive (приватные сети, AWS-метадата, DNS-rebinding, open-redirect, 1 MB cap), Codex round 1-9.
  • Битый/пустой JSON → graceful 422, ни одного 500.
  • Auth-отбой: запросы без токена → 400, bogus UUID → 400/404, данные не текут.
  • i18n RU↔EN на всех страницах — вёрстка не поехала ни на десктопе, ни на мобиле 390×844.
  • SSG /jobs рендерится: «1–12 из 49801», карточки с деталями, пагинация.

Рекомендация — мини-спринт «полировка» перед анонсом

Ничего переписывать не нужно. До анонса достаточно закрыть P1 + самые заметные P2:

  1. Алиасы (BUG-01) — зарегистрировать register_profile/delete_profile как deprecated-обёртки с deprecation_warning, как и планировалось в Спринте 2.
  2. REST-twin удаления работодателя (BUG-02) — добавить DELETE /api/v1/employer/profile в пару к кандидатскому.
  3. 3 правки в skill.md — предупреждение про draft-gate (BUG-03), proof_url в шагах Track A (BUG-08); заодно поправить consent-hint (BUG-06).
  4. Дедуп профилей (BUG-05) — уникальность по proof_url/email, чтобы не засорять матчинг.
  5. Косметика — бампнуть serverInfo.version, валидация language, синхрон title, кнопка support на главной.

Параллельно: анонс не раньше, чем закроется юридический блок (зона Дениса — Условия / Политика обработки данных).
