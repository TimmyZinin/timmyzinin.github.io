02Финальное решение
ЧТО ДЕЛАЕМ В MVP, ЧТО В v2

Три решения принято.

Решение 1 из 3

Сколько вопросов — 150

Не 20 (слишком скромно — теряем wow-момент M2M-опроса). Не 300 (approval fatigue, галлюцинации Claude у дальних полей).

  • 30 hard — matchable SQL-фильтры: стек, уровень, floor, timezone, формат, остальные базовые.
  • 120 soft — богатый контекст: детальная история работы, контакты, соцсети, style, intent. Не участвуют в SQL-матчинге MVP, но наполняют резюме и граф.
  • Топ-10 critical (seniority, years, comp, floor, shipped_products) — с обязательным provenance (цитата из MD + confidence). Галлюцинации отсекаются на approval-экране.
Решение 2 из 3

Резюме оставляем, но генерим автоматом

Polywork закрылся в 2024, пытаясь продать «нет резюме — есть activity feed». Аудитория в найме хочет видеть CV-как-документ. Отказ = провал по прецеденту.

  • CV генерится из 150 ответов + observed_facts. SSG-страница на mcphire.com/cv/<profile_id> с unguessable 8-char slug.
  • Фото, соцсети, опыт, артефакты, блок «связан с / работал над» из графа.
  • Юзер не пишет CV руками — только approve автосгенерированный. Перегенерация через повторный /mcphire-join.
  • CV = артефакт, который забирает себе. Даже если матчей ноль — есть HTML-страница для шаринга. Каждый такой share = органический трафик.
Решение 3 из 3

Граф знаний — минимальный в MVP, полноценный в v2

Граф в чистом виде — правильное направление, но не для старта. Сложность neo4j / semantic web / entity resolution сжирает недели, при этом matching до 500+ профилей всё равно упирается в sparsity.

  • Silent collection в MVP: Claude параллельно с ответами пишет observed facts на сервер в формате subject-predicate-object (worked_at, contributed_to, shipped, mentored).
  • Approval-экран показывает triples юзеру. Только approved_by_user=true идут в публичный CV.
  • В MVP граф виден как секции «связан с» / «работал над» / «знает» на CV-странице. Plain tables в PG, не neo4j.
  • В v2 (~месяц 2-3): когда 200+ профилей — строим полноценный граф на базе тех же observed_facts + entity resolution. Переход без перезаписи данных.

04Панель — короткие выводы
ПЯТЬ РОЛЕЙ · ПЯТЬ УГЛОВ АТАКИ

Что сказал каждый.

Founder

100 — слишком мало, 150 — sweet spot

Для маркетинга «мой Claude ответил за меня 20 вопросов» слабо. 150 — достаточно для wow, но не утомляет на approval. CV-артефакт важнее графа на старте.

Matching Engineer

Граф в MVP = over-engineering

До 500 профилей граф страдает от sparsity. SQL JSONB overlap покрывает 100% кейсов. Observed_facts — как sidecar, не как основной store.

Privacy Analyst

150 — верхняя граница без нарушений

Не спрашиваем: возраст, пол, гражданство, семейный статус, здоровье, религия. Все остальные 150 — legitimate. Provenance защищает от hallucinated claims на критичных полях.

Adversarial QA

Graph hallucinations хуже form hallucinations

Claude выдумает «работал под Илоном Маском», triple попадёт в публичный граф, никто не заметит среди 300 edges. Поэтому — approved_by_user обязателен для каждого triple.

Researcher

Polywork — предостережение

Пытались «нет резюме», 50+ activity-вопросов, закрылись в 2024. Urged: CV остаётся, proof_url anchor обязателен. Граф — overlay, не replacement.

UX / Philosopher

Авторезюме = viral loop

Люди шарят CV-ссылку в TG, в посты, друзьям. Каждый share = органический трафик. Match-only loop этого не даёт. Граф тоже — abstract, не виден в чате.

06Фазы
ЧТО КОГДА И ПОЧЕМУ В ТАКОМ ПОРЯДКЕ

Три фазы, три pitch-момента.

I
MVP · 14 дней
hypothesis test
150 вопросов, CV автосгенерируется, observed_facts пишутся silent. Pitch в LinkedIn: «мой Claude сам собрал мне резюме за 3 минуты, вот ссылка». Первый wow.
II
v1.5 · мес 1-3
growth loop
Собираем 200+ профилей. Buyer-сторона. Embedding-слой для soft-полей. Граф видимый юзеру на /me. Pitch #2: «теперь твой профиль — не форма, а knowledge graph».
III
v2 · мес 3+
scale
Полноценный граф (neo4j / apache age). Multi-hop queries для buyer-агентов («кто работал с ex-Anthropic staff на LangGraph»). M2M-переговоры. Pitch #3: «job board умер, есть граф».
